# 基于 FSOD-VFM 的遥感多类别目标检测比赛介绍与结果分析

本文档面向比赛汇报，基于当前项目代码、FSOD-VFM 论文、比赛评分方案、比赛数据集统计以及 `results/` 目录下已有实验结果整理。需要特别说明的是：当前项目已经按照老师解释后的口径，在 `race_metric.py` 中采用“25 个子类严格匹配、整体汇总计算 Recall/FDR”的评价方式；早期结果文件中也保留了“三大类宽松匹配”的历史结果，二者可以用于解释为什么指标会出现明显下降。

## 1. 比赛需要解决的问题和目标

本比赛的任务是面向光学遥感图像中的陆上时敏目标检测识别。模型需要在测试图像中完成两件事：

1. 正确定位目标：给出目标的检测框，并且与真实框达到规定 IoU 阈值；
2. 正确识别类别：不仅要知道是飞机、舰船或车辆，还要识别到具体子类。

从最终评价角度看，一个预测框想被记为 TP，需要同时满足：

- 与真值框属于同一张图像，即 `image_id` 相同；
- 预测子类 `category_id` 与真值子类完全一致；
- IoU 达到阈值，车辆为 0.35，其他目标为 0.50；
- 该真值框此前没有被更高分预测框匹配。

如果框的位置正确，但子类预测错误，例如真实是 `A16_FA-18`，预测为 `A5_F-16`，则在当前严格口径下不能算 TP，而是该预测框记为 FP，对应真实目标仍然记为 FN。

### 2.2 图像与多目标特点

验证集图像主要是遥感切片，当前检查到的图像尺寸以 800×800 为主：896 张验证图像中，551 张为 800×800；其余图像尺寸在约 672 到 1216 像素之间变化。

每张图像可能包含多个目标。验证集中：

- 每张图目标数最少 1 个，最多 34 个；
- 平均每张图约 4.69 个目标；
- 193 张图只有 1 个目标，但也有图像包含十几个甚至二十多个目标；
- 大多数图像只包含 1 个子类，但也有图像同时包含 2 到 5 个子类。

这意味着模型不仅要检测单目标，还要处理同一张图像中的多目标、密集目标以及可能同时出现的多个子类。

## 3. FSOD-VFM 方法流程

FSOD-VFM 论文的核心思想是利用视觉基础模型完成训练自由或少训练的 few-shot object detection。论文中方法由三个核心组件构成：

- UPN：Universal Proposal Network，用于类别无关的候选框生成；
- SAM2：用于根据候选框得到更准确的目标区域 mask；
- DINOv2：用于提取目标区域特征，并与支持集原型进行匹配；
- Graph Diffusion：用于对候选框分数进行重加权，抑制局部碎片框和冗余框。

当前项目的实现流程与论文描述基本对应。

### 3.1 支持集原型构建

在支持集阶段，代码读取带标注的少样本或 all-shot support JSON。对于每个支持集目标：

1. 读取对应图像；
2. 使用 DINOv2 提取整图特征图；
3. 使用 SAM2 根据真实 bbox 得到目标 mask；
4. 将 mask resize 到 DINOv2 特征图大小；
5. 对 mask 覆盖区域做平均池化，得到该实例的特征向量；
6. 按子类聚合实例特征，计算每个子类的类别原型。

在当前代码中，这部分主要位于 `support_util.py`。`extract_support_features` 会为每个子类收集实例特征，之后 `compute_prototype_weights` 得到最终 `proto_feat` 和 `proto_cls`。对于本比赛，原型是按 25 个子类构建的，而不是只按飞机、舰船、车辆三个大类构建。

项目中还加入了原型缓存机制。当前 `/mnt/data/wangzijian/FSOD-VFM-Public/fsod_cache` 下已有 vitb14 和 vitl14 的 support instance cache 与 prototype cache，因此再次运行 all-shot 推理时通常不需要重新提取支持集特征和计算原型，除非修改了 support JSON、backbone 配置、SAM2 配置，或者显式设置 rebuild。

### 3.2 查询图像推理

在验证集或测试集推理阶段，当前代码流程如下：

1. 对每张 query image，UPN 生成候选框；
2. 先按 UPN 分数排序，当前每张图最多保留 100 个候选框；
3. 对每个候选框，使用 SAM2 预测 mask；
4. 用 mask 在 DINOv2 特征图上做区域平均，得到 query instance feature；
5. 将该特征与 25 个子类原型做相似度计算；
6. 取相似度最高的子类作为预测 `category_id`；
7. 使用图扩散对同类候选框的分数进行衰减或重加权；
8. 每张图最终仍保留 top 100 预测框，保存为 COCO 风格预测 JSON。

当前代码中，类别预测核心逻辑是：

```python
sims = feat_vec @ proto_feat
top_score, top_cls = torch.max(sims, dim=1)
cat_id = name_to_id.get(proto_cls[top_cls[0].item()])
```

也就是说，最终类别来自 query 特征与 25 个子类原型的最大相似度。

### 3.3 图扩散的作用

FSOD-VFM 论文指出，UPN 虽然能产生较高召回的候选框，但容易产生 over-fragmentation，也就是把一个完整目标拆成很多局部候选框。这会造成大量碎片框和虚警框。

Graph Diffusion 的作用不是重新分类，而是根据同类候选框之间的 mask 关系构建图，对候选框分数做重加权。直观理解是：

- 更完整、更稳定的目标框应保留较高分；
- 局部碎片框、与其他框关系异常的框应被降分；
- 它更像一种 soft reweighting，而不是 NMS 那种直接删除框的硬规则。

在当前项目中，图扩散不会改变 `category_id`，主要影响 `score` 排序。因此如果一个候选框已经被分错子类，图扩散通常不能把它修正为正确子类。

## 4. 当前比赛场景中的主要挑战

### 4.1 类别极不均衡

数据集存在明显长尾分布。验证集中：

- `HM` 只有 4 个目标；
- `LQS` 只有 10 个目标；
- `A18_KC-10` 只有 56 个目标；
- `A16_FA-18` 有 467 个目标；
- `A14_KC-135` 有 322 个目标。

训练集中也类似：`HM` 只有 13 个目标，而 `MS` 有 1695 个目标，`A16_FA-18` 有 1680 个目标。这种不均衡会影响原型质量：少样本子类的原型可能不稳定，而样本多的类别更容易形成稳定原型。

### 4.2 同一大类内部子类相似度高

比赛要求区分 20 种飞机子类，以及 4 种舰船子类。遥感图像中目标尺度较小、视角固定、纹理信息有限，不同飞机型号或舰船型号在 DINOv2 特征空间中可能非常接近。

因此，模型可能已经框到了飞机，但把 `A16_FA-18` 预测成 `A5_F-16`，或者把 `HM` 预测成 `LQS`。在三大类宽松口径下，这类预测可能被视为正确；但在严格子类口径下，这会同时造成一个 FP 和一个 FN。

### 4.3 目标尺度差异与小目标问题

遥感图像中目标尺度变化大，车辆和部分舰船/飞机目标在切片中可能很小。小目标区域包含的有效像素少，SAM2 mask 和 DINOv2 区域特征更容易受到背景影响，导致原型匹配不稳定。

### 4.4 每图 top100 导致 FDR 被放大

当前 FSOD-VFM 代码延续了论文/AP 评估习惯：每张图输出 top100 预测框。验证集有 896 张图，因此预测文件固定有：

```text
896 张图 × 每图 100 框 = 89600 个预测框
```

但验证集真值只有 4206 个目标。即使位置召回较高，剩余大量没有匹配到真值的预测框也都会成为 FP。因此在比赛 FDR 口径下，全量 top100 输出会天然造成很高虚警率。

降低预测框数量、设置 score threshold 或使用 NMS 可以降低 FDR，但也会降低 Recall。当前严格子类实验中，单纯阈值过滤并不能达标，因为 Recall 本身已经受子类错误限制。

## 5. 评价指标计算流程

比赛评分方案规定：

```text
Recall = TP / (TP + FN)
FDR = FP / (FP + TP)
```

其中：

- TP：预测结果与某个真实目标成功匹配，且该真实目标此前未被其他预测框匹配；
- FP：预测报出了目标，但未能匹配任何真实目标，或属于重复检测框；
- FN：真实目标未被任何预测结果成功匹配。

匹配时按预测 score 从高到低排序。每个预测框最多匹配一个真实框，每个真实框最多被一个预测框匹配。多个预测框匹配同一真实目标时，最高分预测框计为 TP，其余计为 FP。

IoU 公式为：

```text
IoU = Area(B_pred ∩ B_gt) / Area(B_pred ∪ B_gt)
```

当前代码中，车辆目标 IoU 阈值为 0.35，舰船和飞机为 0.50。

### 5.1 三大类宽松口径

早期项目结果采用过一种宽松口径：只要求预测目标与真值目标属于同一大类，例如都属于飞机，即使具体飞机型号不同，也可能被算作匹配成功。这种口径更接近“三大类检测”的理解。

该口径下，模型主要考察是否检测出飞机、舰船、车辆这三个大类，子类混淆不会受到严厉惩罚。因此 Recall 会明显更高。

### 5.2 组委会/老师解释后的严格子类口径

结合老师解释“所有类型合并计算，只有子类正确才认定正确”，当前正确理解是：

- 最终只计算一组整体 Recall/FDR；
- 不是分别计算 25 个子类再取平均；
- 也不是分别计算飞机、舰船、车辆后取平均；
- 而是将 25 个子类产生的 TP、FP、FN 全部汇总；
- 但 TP 判断时必须要求 `category_id` 与真值完全一致。

因此当前严格口径为：

```text
同一 image_id
+ category_id 完全相同
+ IoU 达到阈值
+ GT 尚未被更高分预测匹配
= TP
```

整体指标为：

```text
TP_all = 所有 25 个子类 TP 汇总
FP_all = 所有 25 个子类 FP 汇总
FN_all = 所有 25 个子类 FN 汇总

Recall = TP_all / (TP_all + FN_all)
FDR = FP_all / (TP_all + FP_all)
```

这也是当前 `race_metric.py` 的实现方式。

## 6. 结果分析

### 6.1 三大类宽松口径结果

以 `results/race_dinov2_vitl14_all/race_eval_summary_20260620_142347.json` 为例，宽松三大类口径下：

| 实验 | 口径 | 阈值来源 | 阈值 | predictions | TP | FP | FN | Recall | FDR |
|---|---|---|---:|---:|---:|---:|---:|---:|---:|
| vitl14 宽松三大类 | all predictions | 无 | - | 89600 | 4130 | 85470 | 76 | 98.19% | 95.39% |
| vitl14 宽松三大类 | selected threshold | automatic_balanced_passing_point | 0.854516745 | 4456 | 3713 | 743 | 493 | 88.28% | 16.67% |

可以看出，如果不设置阈值，虽然 Recall 高达 98.19%，但由于每张图输出 top100，总预测框数达到 89600，FP 高达 85470，FDR 为 95.39%。这说明全量 top100 不能直接作为比赛提交结果。

当采用自动选择的阈值 0.854516745 后，预测框数从 89600 降到 4456，FP 从 85470 降到 743，FDR 降到 16.67%，Recall 仍有 88.28%。在宽松口径下，该阈值可以同时满足 Recall ≥ 85% 和 FDR ≤ 20%。

宽松口径下各大类 all predictions 指标如下：

| 大类 | GT | predictions | TP | FP | FN | Recall | FDR |
|---|---:|---:|---:|---:|---:|---:|---:|
| ship | 487 | 42591 | 429 | 42162 | 58 | 88.09% | 98.99% |
| aircraft | 3645 | 13278 | 3611 | 9667 | 34 | 99.07% | 72.80% |
| vehicle | 74 | 33731 | 70 | 33661 | 4 | 94.59% | 99.79% |

这里可以看到：飞机大类的 Recall 很高，说明 UPN + 原型匹配对飞机目标的位置覆盖能力较强；但舰船和车辆预测数量过多，FDR 极高，尤其车辆只有 74 个 GT，却产生了 33731 个 vehicle 预测。

### 6.2 严格子类口径结果

以 `results/race_dinov2_vitl14_all_0623/race_eval_summary_20260623_220156.json` 为例，严格子类口径下：

| 实验 | 口径 | 阈值来源 | 阈值 | predictions | TP | FP | FN | Recall | FDR |
|---|---|---|---:|---:|---:|---:|---:|---:|---:|
| vitl14 严格子类 | all predictions | 无 | - | 89600 | 2966 | 86634 | 1240 | 70.52% | 96.69% |
| vitl14 严格子类 | selected threshold | not_available | - | - | - | - | - | - | - |

严格口径下，即使使用所有 top100 预测，Recall 也只有 70.52%，说明许多目标虽然可能被框到，但子类没有预测正确。与宽松口径相比：

```text
宽松三大类：TP=4130, FN=76, Recall=98.19%
严格子类：TP=2966, FN=1240, Recall=70.52%
```

TP 减少了 1164 个，FN 增加了 1164 个。这部分差距主要来自“位置可能匹配，但子类不一致”的预测。例如真实是某一型号飞机，预测成另一型号飞机；宽松大类口径下可能算对，严格子类口径下则不算对。

严格口径下各大类 all predictions 指标如下：

| 大类 | GT | predictions | TP | FP | FN | Recall | FDR |
|---|---:|---:|---:|---:|---:|---:|---:|

| aircraft | 3645 | 13278 | 2537 | 10741 | 1108 | 69.60% | 80.89% |

可以看到，严格子类口径对飞机影响最大：飞机 Recall 从宽松口径的 99.07% 降到 69.60%。这说明飞机目标的位置检测能力并不差，但 20 个飞机子类之间的细粒度分类是主要瓶颈。

### 6.5 类别预测分布偏置

预测文件中存在明显类别偏置。以 vitl14 预测文件为例，89600 个预测框中数量最多的类别为：

| category_id | 子类 | GT 数 | 预测数 | 现象 |
|---:|---|---:|---:|---|
| 25 | FSC | 74 | 33731 | 车辆预测严重过多 |
| 1 | HM | 4 | 27269 | 极少样本舰船子类被大量误预测 |
| 4 | MS | 299 | 7798 | 舰船预测偏多 |
| 2 | LQS | 10 | 5855 | 极少样本舰船子类被大量误预测 |

这说明当前原型匹配在某些子类上存在明显吸附效应：大量候选框被分到 `FSC`、`HM`、`MS`、`LQS` 等类别。严格子类评价下，这会显著增加 FP，并导致其他子类 GT 无法被正确匹配。

### 6.6 关于推理时间

比赛要求单幅 10000×10000 图像推理时间 ≤ 20 秒。当前项目代码中，推理时间统计位于 `metric.py`：

```text
Average time per current validation tile
Estimated 10k sequential time (144 tiles)
Estimated 10k sequential time (169 tiles)
```

也就是说，当前代码不是直接在完整 10000×10000 大图上计时，而是在验证集切片图像上统计平均时间，再按 10000×10000 大图可切成多少个 tile 进行估算。验证集图像大多为 800×800，因此估算时使用了 144 或 169 个 tile 的倍数。

需要客观说明的是：当前 `results/*.json` 文件没有保存 vitb14/vitl14 的实际推理时间日志，代码只是将时间打印到终端。因此，目前已有结果文件可以支持精度指标分析，但不能从 JSON 中恢复当时的具体推理秒数。若后续需要在汇报中给出 vitb14 和 vitl14 的具体时间，建议重新运行并保存终端日志，例如：

```bash
GPU=1 \
SHOT=all \
MODEL_VERSION=dinov2_vitl14 \
OUTPUT_DIR=./results/race_dinov2_vitl14_all_time \
bash run_scripts/run_race.sh 2>&1 | tee results/race_dinov2_vitl14_all_time/run.log
```

然后从 `run.log` 中读取 `Average time per current validation tile` 和估算的 10k 图像耗时。一般来说，vitl14 特征更强但计算量更大，vitb14 速度更快但严格子类 Recall 较低；实际取舍需要结合保存后的时间日志与严格子类精度共同判断。

## 7. 综合结论

1. 本比赛本质上不是简单的大类目标检测，而是遥感多目标细粒度检测识别：不仅要框出目标，还要识别 25 个具体子类。

2. FSOD-VFM 的 UPN + SAM2 + DINOv2 + Graph Diffusion 流程适合少样本目标检测，UPN 提供高召回候选框，SAM2 提供目标区域 mask，DINOv2 提供泛化特征，图扩散用于降低碎片框分数。

3. 当前宽松三大类口径下，vitl14 在阈值 0.8545 附近可以达到 Recall 88.28%、FDR 16.67%，说明候选框定位和大类级检测能力较好。

4. 当评价变为严格 25 子类口径后，vitl14 全量 Recall 从 98.19% 降到 70.52%，TP 从 4130 降到 2966，说明主要瓶颈转移到细粒度子类识别。

5. 全量 top100 输出导致 FDR 极高。896 张图每张输出 100 框，共 89600 个预测，而 GT 只有 4206 个；大量预测框无法匹配真值，因此直接成为 FP。

6. 仅靠 score threshold 可以降低 FDR，但会同步降低 Recall。严格子类口径下，当前结果找不到同时满足 Recall ≥ 85% 和 FDR ≤ 20% 的阈值。

7. 类别不均衡和子类相似是当前主要挑战。尤其是 `HM`、`LQS`、`FSC` 等少样本或易混类别在预测中出现明显偏置，导致 FP 大量增加。

8. 后续优化方向应重点放在：提升子类判别能力、改善原型构建方式、校准类别分数、减少每图最终输出框数量、引入更适合比赛 FDR 的后处理策略，而不是只依赖阈值过滤。

## 8. 可用于汇报的一句话总结

本项目使用 FSOD-VFM 将视觉基础模型迁移到遥感少样本目标检测任务中。方法在候选框定位上具有较强召回能力，但比赛最终要求 25 个子类严格正确，因此细粒度子类识别成为主要瓶颈；同时每图 top100 的 AP 风格输出在比赛 FDR 口径下会产生大量 FP。当前结果表明，宽松大类口径下模型可以通过阈值达到指标，但严格子类口径下 Recall 和 FDR 均未达标，后续需要从子类原型判别、分数校准和最终预测过滤三方面继续优化。
